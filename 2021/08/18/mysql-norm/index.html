MySQL 规范

规范的目的

  1. 提高数据库系统的处理效率
  2. 避免数据库不必要的死锁及资源浪费
  3. 提高应用系统的数据库升级方便

约定

  1. 简单应用 MySQL，只做存储，尽量不做运算（计算由程序端 CPU 处理）

    e.g. 避免 md5（）/ order by rand()

  2. 控制表/库数据量

    • 表字段数少而精：IO高效，全表遍历，表修复快，提高并发，alter table 快
    • 单表字段数上限控制在 20-50 个

    • 单表1g体积，500w行评估

      • 顺序读1g文件需n秒
      • 单行不超200Byte
      • 单表不超50个纯INT字段
      • 单表不超20个CHAR(10)字段
    • 合理分表不超载：USERID/DATE/AREA
    • 建议单库不超过 300-400 个表

    • 单库容量：

      • SAS盘Raid 10 控制在500G左右
      • PCI-E卡，可以控制在卡的大小80%左右
      • 日志或是历史库除外

  3. 平衡范式与冗余

    • 效率优先、提升性能
    • 适当时牺牲范式、加入冗余
  4. 拒绝3B — 大SQL、大事务、大批量

  5. 必须有主键：AUTO_INCREMENT/全局 ID

    • 建议使用 INT/BIGINT 并且自增做为主键，顺序 insert 效率更高，表空间碎片率更低
    • 主键避免采用字符型，如VARCHAR/CHAR/UUID，会导致原本可以顺序写入的请求变成随机写入，效率更低

索引

Def

  • 二级索引：非主键索引（聚簇索引）/ 普通索引，只含索引字段+主键，需要回表才能取到完整记录数据

  • 覆盖索引：查询所需数据就在普通索引/非聚簇索引树上，无需回表。

  • 索引（条件）下推（ICP, index condition pushdown）：MySQL5.6 引入，失效索引（索引功能由存储引擎提供）也能在引擎层中进行条件判断（无需回表到 server 层进行判断），减少了回表进而提高查询效率。

  • 最左匹配 ：在联合索引中，只有左边的字段被用到，右边的才能够被使用到。

索引代价

时间：

  • B+树维护

    键索引的B+树的每一个节点内的记录都是按照主键值由小到大的顺序，采用单向链表的方式进行连接。

    增删数据时可能引起主键页内重排序，甚至是数据页的分类与回收。

  • 生成执行计划

    优化器会计算每个索引的搜索成本比较生成最优执行计划，索引过多增加该过程耗时，降低 sql 执行效率。

空间：一个索引 == 一颗B+树，B+树的每个节点都是一个数据页，每个数据页默认会占用16KB的磁盘空间，因此，B+树节点越多 -> 数据页越多 -> 磁盘占用越大。

回表代价

原则：

  1. 能不回就不回
  2. 必须回就减少次数

索引创建原则

  1. 索引能不加则不加（单表索引数 <= 5）

    • 优先考虑联合索引（可用于覆盖索引）
    • 小表不使用（全表扫描性能差距不大，节约空间）
    • 区分度低（重复值多）的字段不使用
    • 随机无序字段不使用（e.g. 身份证、UUID）

  2. 常见优先场景：

    • ID 类字段
    • 搜索/排序/分组，即WHERE / JOIN / GROUP BY / ORDER BY 常用字段

  3. 充分设置联合索引，且区分度大的字段靠前

    区分度：唯一值个数 / 表长。

    降低索引树的高度，查询时减少磁盘 IO 次数，提高效率。

  4. 频繁更新字段不加索引

    重建索引树的成本高。

  5. 字符字段必须建前缀索引，且禁作主键
  6. EXPLAIN 查看 SQL 执行计划的索引使用情况

  7. 索引命名：

    • 非唯一索引按照“i字段名称字段名称[_字段名]”索引命名
    • 唯一索引按照“u字段名称字段名称[_字段名]”

索引使用原则

  1. 充分利用唯一索引

    • 只需找到一条目标记录即返回
    • 若由 ≤3 个字段组成，且字段都是整数型，使用唯一键作为主键。

  2. 索引字段作为单纯左值

    索引字段使用运算表达式或函数会导致索引失效

  3. 避免隐式类型转换

    字段操作类型不匹配会引起隐式类型转换，导致索引失效

    数据库隐式类型转换方式检测 select '10' > 9;：

    • 1 — 字符串转数字
    • 0 — 数字转字符串

  4. 避免反向查询

    反向查询：NOT、!=、<>、!<、!>、NOT EXISTS、NOT IN、NOT LIKE 等 — 索引失效

  5. 遵循 最左匹配 原则

    当联合索引中的某字段遇到范围查询 BWTWEEN时，就停止某字段右边索引的匹配，右边部分字段索引失效

    e.g. 如联合索引 KEY idx_abc(a, b, c) 遇上条件 WHERE a BWTWEEN (1, 10) AND b = 1 AND c = 2，实际使用到的索引的字段只有 a。可通过 EXPLAIN 中的 key_length 确认。

  6. or 条件列中存在非索引列，or 条件中的索引列也会索引失效

SQL

  1. 多条简单 SQL >> 单条大 SQL

    • 减少锁表时间，特别是 MyISAM
    • 多条多 CPU > 单条单 CPU：充分使用硬件，速度快，降低阻塞
    • 简单 SQL 缓存命中率更高

  2. 保持事务(连接)短小

    • 事务/连接使用原则：即开即用，用完即关
    • 带锁 SQL 尽量放事务后期，减少占用时间
    • 与事务无关操作放到事务外面，减少锁资源的占用
    • 不破坏一致性前提下，使用多个短事务代替长事务
  3. 线上 OLTP 尽可能避免使用 SP/TRIG/FUNC/UDF，由客户端处理
  4. 避免 SELECT *，降低 CPU、内存、IO、网络带宽的消耗

  5. 改写 OR O(n)

    • 同一字段 — IN，效率 O(n) → O(Log n)，但控制 IN 集合＜200
    • 不同字段 — 分字段查询再 UNION

  6. 避免反向查询、前缀模糊查询

    • 反向查询：NOT、!=、<>、!<、!>、NOT EXISTS、NOT IN、NOT LIKE 等 — 无法使用索引
    • LIKE% 前缀模糊 — 导致全表扫描

  7. 减少 COUNT(*)

    • 开销大，尽量不用/少用/空闲用

    • 计数统计代替

      • 实时统计：用 memcache，双向更新，凌晨跑基准
      • 非实时统计：尽量用单独统计表，定期重算

  8. LIMIT 利用主键索引高效分页

  9. 用 UNION ALL 而非 UNION（存在去重开销）

  10. 分解 JOIN联接保证高并发

    • 高并发 DB 不建议进行 > 2个表的 JOIN

    • 适当分解 JOIN 联接保证高并发

      1. 先缓存高频 join 数据表

      2. 分步多次查询再 ID IN

  11. 去重用 GROUP BY

    • 有序：GROUP BY
    • 无序：GROUP BY <group_key> ORDER BY NULL

  12. 字段间的比较尽量保持数据类型相同

    • 避免隐式类型转换 — 不同数值类型比较，“小往大看齐”，即小类型转为大类型再比较

      在non-strict mode下，MySQL会自动帮你把字符串转换成整形，但是如果数值超出了范围，转换就会失败，所以MySQL就按照字符串来处理，因此不能使用索引。而从explain的结果上，并没有表现出这样的差别。

    • 隐式类型转换无法使用索引，相当于应用了函数

  13. 减少排序

    • 充分利用索引和 GROUP BY
    • 避免 UNION ALL 和 DISTINCT
  14. WHERE中区分度大的条件靠前，具体看执行计划

  15. 去掉无意义的连接用条件：1=1，2>1，1<2 等

    MyBatis 或 ORM 对象关系映射 替代 动态添加 SQL 标准化。

  16. 使用 LOAD DATA 导数据

    • 成批装载比单行装载更快，不需要每次刷新缓存
    • 无索引时装载比索引装载更快
    • INSERT values ,values，values 减少索引刷新
    • LOAD DATA 比 INSERT 快约20倍
    • 尽量不用 INSERT ... SELECT：1. 延迟；2. 同步出错

  17. UPDATE 时禁止将“,”写成“and”

  18. MySQL 不支持对一张表同时进行UPDATE 和 SELECT（在同一 sql 中对同一张表先SELECT后UPDATE）

    常见场景：

    解决核心：使 sql 不对同一张表进行操作/另命名

    1. select from 嵌套一层子查询

      查询内容另作一张表再 select出来。

    2. inner join 表另名

  19. 尽量使用 主键 或 WHERE 字段加索引 进行 UPDATE和 DELETE

    UPDATE和 DELETE是表锁，影响性能。

  20. 避免不必要的查询和数据库的交互

  21. 打散大批量更新

    • 避开高峰期，大批量更新凌晨操作

    • 白天设置上限默认为 N 条/秒

其余约定

  1. 命名规范

    • 库名/表名统一小写

      MySQL库表大小写敏感，字段名的大小写不敏感。

    • 库名用缩写（2~7个字母）

      e.g. students course — sc

    • 避免用保留字命名

  2. 统一字符集为 UTF8

    • 校对规则：utf8_general_ci
    • 乱码：SET NAMES UTF8
    • 存储emoji字符：utf8mb4

  3. 每张表三个必加字段：

    • aid（int/bigint unsigned类型，自增长列，并且作为主键）
    • create_time（timestamp或int unsigned）
    • update_time（和create_time 相同）

  4. 不在程序端显式加锁

    • 外部锁对数据库不可控，高并发时是灾难，极难调试和排查
    • 采用事务解决并发更新等一致性问题
    • 相对值修改在 Commit 前二次较验冲突

  5. 尽量避免使用子查询

    • 子查询大部分情况优化较差，最好使用 DBA 确认的子查询
    • 用 JOIN 代替 WHERE IN 的子查询

  6. 敏感数据加密再存储

    严禁明文存储用户密码、身份证、信用卡号（信用卡PIN码）等核心机密数据，务必先行加密，避免明文存储

  7. 隔离线上线下

    线上连线上，线下连线下。

    实时数据用 real 库，模拟环境用 sim 库，测试用 qa 库，开发用 dev 库。

防后悔

  1. 操作 delete / update

    • 先 where……select 确认无误后再执行
    • 备份数据

    • 加 limit

      • 降低一次误删的代价 — 通过 binlog 快速回滚
      • 可能提高效率 — 待删除记录 <10 条，limit 10 删完即返回，不 limit 继续扫描直至不符合条件
      • 一次删除数据量大，可能导致大量纪录被锁或 CPU 打满
  2. 变更 SQL 应有明细步骤和回滚方案，测试、review再上产线
  3. SQL 命令行修改数据养成 begin……end; 的习惯，方便回滚
